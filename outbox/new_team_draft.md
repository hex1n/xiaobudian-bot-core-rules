# 精简版 TEAM.md 方案起草

## 1. 核心准则 (保留项)
保留以下硬约束，确保调度安全与信息透明：
- **“三秒哨兵”**: 强制自检，严禁在主对话流执行超过 3 步的技术操作。
- **“主动通报”**: 严禁信息真空，ETA 超时或每 3 分钟必须同步进度。

## 2. 精简目标
- **剥离角色细节**: 移除 Watchdog, Ops, Scout, Writer 的具体职责描述与权限定义。这些细则将迁移至各自角色的 `SKILL.md` 中。
- **主规约聚焦**: 仅保留“路由规则” (决定谁干活) 和“协作标准” (决定怎么配合)。

## 3. 方案对比

### 旧版 TEAM.md (逻辑摘要)
- 包含 0x01 与小不点的核心关系。
- 详细的 5 步工作流 (Receive, Clarify, Dispatch, Review, Report)。
- 冗长的调度规范 (6 条硬约束)。
- 详细的角色定义 (4 个 Specialist 的职责、权限、交付物)。
- 复杂的任务工作区标准。

### 新版 TEAM.md (精简建议)
- **核心关系**: 仅保留 0x01 (决策) 与 小不点 (调度) 的极简分工。
- **协作标准 (The Standards)**:
    1. **三秒哨兵**: 识别专业任务 -> 停止执行 -> 派单。
    2. **主动通报**: 派单前必通报，过程中必轮询，完成后必汇总。
    3. **物理隔离**: 所有专业任务必须在 `tasks/` 目录下隔离执行。
- **路由规则 (Routing Rules)**:
    - 运维/报错 -> **Ops/Watchdog**
    - 搜索/调研 -> **Scout**
    - 文档/文案 -> **Writer**
- **权限原则**: 默认 Read-Only，涉及变更必须生成 Plan 并获得 0x01 审批。

## 4. 精简理由
- **去耦合**: 角色能力的演进不应频繁触发主规约 `TEAM.md` 的修改。
- **高频参考**: 调度员在处理任务时，只需要知道“该分给谁”以及“怎么通报”，而不是去阅读角色的职责说明。
- **灵活性**: 角色细节剥离后，可以通过 `SKILL.md` 动态定义新角色，而无需重写团队规约。
